## Bugs Found

### BUG-1 (MEDIUM): isExampleLine bypasses structural token detection on lines with template variables
- **File**: `src/rules/sec-rules.ts:35`
- **Description**: `isExampleLine()` was applied at the line level BEFORE any pattern matching. The function returns true if the line contains `${` (among other markers). This meant ANY line with a shell variable like `${HOME}`, `${USER}`, `${TOKEN_VAR}`, or `${AWS_PROFILE}` was treated as a documentation example and skipped entirely -- including lines with real, structurally-identifiable tokens like `AKIAIOSFODNN7EXAMPLE`, `ghp_...`, or `sk-ant-...`. The code already had a per-match structural token bypass (line 44: `isStructuralToken` check) but it was unreachable because `isExampleLine` skipped the line before the pattern loop.
- **Root cause**: Defense-in-depth ordering error. The coarse line-level filter (`isExampleLine`) ran before the fine-grained match-level filter (`isStructuralToken`), defeating the structural token bypass.
- **Impact**: Real AWS access keys, GitHub PATs, Anthropic API keys, Slack tokens, and private keys embedded in lines that also reference shell variables would be silently missed. Common in config documentation like "Set AKIAIOSFODNN7EXAMPLE via ${AWS_PROFILE}".
- **Pattern match**: Similar to "Operation Order Dependency" (R71 bug_patterns) -- a validation is checked before the data it depends on is available.
- **Fix**: Removed line-level `isExampleLine` check. Moved it to per-match level, combined with `isPlaceholder` behind the `!isStructuralToken` guard: `if (!isStructuralToken && (isExampleLine(line) || isPlaceholder(value))) continue;`
- **Regression tests**: +2 tests in `sec-rules.test.ts`:
  1. "should detect structural tokens even on lines with template variables" -- verifies AKIA, ghp_, sk-ant- all detected on `${...}` lines
  2. "should still skip non-structural tokens on example lines" -- verifies generic api_key patterns still filtered on `${...}` lines

### BUG-2 (LOW): Missing prepublishOnly script enables stale dist/ on npm publish
- **File**: `package.json`
- **Description**: No `prepublishOnly` lifecycle script to rebuild dist/ before `npm publish`. Cycle B's BUG-2 was exactly this: source files were modified but dist/ was not rebuilt, shipping stale compiled JS. Without `prepublishOnly: "npm run build"`, the same class of bug can recur any time source changes are followed by `npm publish` without a manual build step.
- **Root cause**: Missing npm lifecycle guard.
- **Impact**: If a contributor modifies source and runs `npm publish`, users receive stale compiled output. Cycle B's BUG-2 demonstrated this exact scenario.
- **Fix**: Added `"prepublishOnly": "npm run build"` to package.json scripts.
- **Regression test**: None needed -- lifecycle script is a process guard, not testable logic.
