## Bugs Found

### BUG-1 (MEDIUM): "info" severity config setting ignored
- **File**: `src/engine.ts:148`
- **Description**: `resolveSeverityFromSetting` checked for "error", "warn", "warning" but NOT "info". When user config set a rule to "info", it fell through to `defaultSeverity` (typically "warning"). Users could not downgrade rules to info severity.
- **Root cause**: Missing condition in switch logic.
- **Additional issue**: The `report()` callback used `diag.severity ?? severity` which meant rules always overrode the config severity because they always provide their own severity. Changed to unconditionally use config-resolved severity.
- **Fix**: Added explicit "info" handling. Changed report callback to always use config severity.
- **Regression test**: `engine.test.ts` - "should honor info severity setting in config"
- **Regression test**: `integration.test.ts` - "should respect info severity in config (not fall through to default)"

### BUG-2 (MEDIUM): --quiet mode does not update summary counts
- **File**: `src/cli.ts:52-57`
- **Description**: `--quiet` filtered diagnostics from `report.results` but `report.totalWarnings`, `report.totalErrors`, `report.totalInfos` remained stale (reflecting unfiltered data). JSON output showed `totalWarnings: 3` with empty `results: []`.
- **Pattern match**: "Filter Applied to Data But Not Metadata" (R75 sessaudit)
- **Fix**: Recompute summary counts from filtered results.
- **Regression test**: `integration.test.ts` - "should properly count diagnostics after quiet-mode filtering"

### BUG-3 (MEDIUM): --quiet mode exit code wrong
- **File**: `src/cli.ts:77-83`
- **Description**: Exit code was based on unfiltered `report.totalWarnings` / `report.totalErrors`. In quiet mode with only warnings (no errors), exit code was 1 even though quiet mode displays nothing. Should exit 0 since no errors are being shown.
- **Fix**: Same as BUG-2 — recomputing counts after filtering fixes exit code logic automatically.
