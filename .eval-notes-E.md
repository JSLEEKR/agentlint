## Bug Found

### BUG-1 (MEDIUM): Doc-drift badge test silently passes due to nested vitest returning 0
- **File**: `src/__tests__/doc-drift.test.ts`
- **Description**: The badge doc-drift test used `execSync("npx vitest list 2>/dev/null | wc -l")` to count actual tests, then compared against the README badge count. However, running `npx vitest list` inside an already-running vitest process (nested execution) returns 0 results instead of the actual test count. The test had an `if (!isNaN(actualCount) && actualCount > 0)` guard that skipped the assertion when actualCount was 0, so the entire badge check was effectively a no-op. The badge could be set to any number (tested with 999) and the test would pass.
- **Root cause**: Nested vitest execution returns 0 results. The guard condition was too permissive -- it should fail when the count method returns 0, not silently skip.
- **Additional issue**: After switching to the `\bit\(/g` regex counting method, the comment in the test itself contained a literal `it(` pattern that the regex matched, inflating the count by 1 (308 vs 307). Fixed by rewording the comment.
- **Impact**: The README badge test count could drift from the actual test count without any test failure. Only the second test (sentence count) provided actual drift protection.
- **Pattern match**: "Silent Skip on Failure" -- a fallback path that silently succeeds instead of signaling an error.
- **Fix**: Replaced `execSync("npx vitest list")` with the same reliable `\bit\(/g` regex counting method already used by the sentence count test. Removed the `execSync` import (now unused). Rewrote the comment to avoid containing a literal `it(` pattern.
- **Regression test**: The existing badge drift test IS the regression test -- it now actually works. Verified by temporarily changing badge to 999 and confirming the test fails.
