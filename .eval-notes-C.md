## Bugs Found

### BUG-1 (MEDIUM): findCommandLine regex-escapes then uses String.includes() -- wrong line numbers
- **File**: `src/rules/sec-rules.ts:177`
- **Description**: `findCommandLine()` was called by `checkJsonHooksForDangerousPatterns()` to find the line number of a dangerous hook command in settings.json. It regex-escaped the command substring (`.` -> `\.`, `$` -> `\$`, `+` -> `\+`, etc.) then used `String.includes()` for matching. `includes()` is a *literal* string match -- the escaped backslashes became literal characters that never matched the actual JSON line content. Any command containing `.`, `$`, `+`, `(`, `)`, `{`, `}`, `|`, `[`, `]`, `\`, `*`, `?`, or `^` would silently fall back to line 1. This affects common commands like `node ./script.js`, `eval $USER`, `chmod +s file`.
- **Root cause**: Confusion between regex escaping (for `new RegExp()`) and literal string matching (for `includes()`). The function name and the `escaped` variable name suggest the author intended regex matching but used `includes()`.
- **Impact**: Diagnostics from `sec-no-dangerous-hooks` and `sec-no-shell-injection` (when checking JSON hook configs) pointed to line 1 instead of the actual line containing the dangerous command. Users investigating reported issues would look at the wrong line.
- **Pattern match**: Similar to "String Prefix Match Without Delimiter" but more precisely a "Regex Escape Applied to Non-Regex Context" pattern.
- **Fix**: Removed the regex escaping. `findCommandLine` now uses the raw substring directly with `includes()`, matching `findApproxLine()` (line 381) which already did this correctly.
- **Regression test**: `sec-rules.test.ts` - "should report correct line number for commands with regex-special chars in JSON" -- creates a multiline settings.json with `chmod +s ./deploy.sh` on line 5 and verifies the diagnostic points to line 5, not line 1.

### BUG-2 (MEDIUM): Unused `ajv` production dependency shipped to users
- **File**: `package.json`
- **Description**: `ajv` (JSON Schema validator, ~800KB) was listed in `dependencies` but never imported or used anywhere in the source code. It ships to every user who `npm install agentlint`, adding dead weight to their `node_modules`.
- **Root cause**: Likely a leftover from a planned SARIF schema validation feature or JSON config schema validation that was never implemented.
- **Impact**: Inflated package size, unnecessary transitive dependency tree, potential supply chain exposure for zero functionality.
- **Fix**: Removed `ajv` from `dependencies` in package.json.
- **Regression test**: None needed -- the dependency was never imported, so removing it cannot break anything.
